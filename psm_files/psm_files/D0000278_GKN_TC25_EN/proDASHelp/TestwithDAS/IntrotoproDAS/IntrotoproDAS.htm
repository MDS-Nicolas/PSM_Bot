Introduction to proDAS

Overview

proDAS is a state-of-the-art, high-performance, Real-Time data acquisition, control, and analysis system that meets the requirements of a modern, multipurpose, engine test facility. proDAS uses a Real-Time Linux operating system to provide a deterministic response to alarms and user inputs. proDAS offers a high degree of user friendliness, flexibility, and programmability. proDAS provides high-resolution graphics with real-time updates and uses an ORACLE database for the storage of test results and calibrations.

The operator can use proDAS to control the information that is recorded during an engine test, control the steps of an engine test, and make test information available for analysis and reporting.

In proDAS, "engine" refers to any turbomachine being tested and includes, but not limited to, all types of gas turbine engines (industrial, marine, military or civil aviation gas turbine engine), compressors, turbines, pumps etc.

proDAS is based on the concept that an engine test consists of configuration information (channel lists, hardware definitions, etc.), measured data (transient and steady-state data recordings), derived performance data, and reports.

proDAS can provide the following features:

  • graphical user interfaces, specifically designed for use by engine operators, calibrators, and test engineers, that incorporate simple point-and-click operation and configuration
  • modular hardware and software architecture and the linking of components using industry standard computers, buses, and protocols
  • true multitasking on multiple CPUs, accomplishing real-time data acquisition and real-time calculations up to 200 hertz
  • real time alarm checking with automatic, programmable actions in response to alarms and "on the fly" alarm limit adjustments
  • user configurable displays that can present engine data in the form of digits, gauges, bar charts, x-y strip charts and graphs with "on the fly" screen editing
  • acquisition of time averaged steady-state data that is stored in a database for analysis
  • acquisition of transient time-based data that is stored for analysis
  • user configurable rotating critical log buffer that continuously monitors a predetermined set of engine parameters
  • continuous logging to automatically record all hardware channel data whenever proDAS is scanning (the availability of this feature will depend on the hardware setup)
  • logging of all system event messages, including operator interactions, alarm conditions, warning messages, error messages, and self-test diagnostics
  • security controlled remote diagnostic access via modem
  • multilevel security that is configurable to meet your requirements
  • point-and-click operation of integrated test procedures
  • assisted calibration routines
  • auto calibration (the availability of this feature will depend on the hardware setup)
  • remote network access, allowing viewing of real-time data across the network

proDAS Architecture

The proDAS system is comprised of:

  • proDAS Computer Systems
  • Hardware Subsystems
  • Software Subsystems
  • Configuration Database
  • Test Results and Sensor Calibration Database (TRSCDB)
  • proDAS Software Utilities
  • Real-Time Data Acquisition Software

proDAS Computer Systems

Each proDAS computer system plays a vital role in the systems architecture. All of the computers are networked together to allow the proDAS software applications to communicate with each other and to communicate with the respective hardware subsystems that have been configured for the engine test.

WIBU keys are used to protect/prevent proDAS software from being copied. A WIBU key is required for each Windows computer and the Linux Host Computer.

The proDAS computer systems include:

  • Linux Host Computer
  • Management GUI Computer
  • multiple Real-Time Display Computers

Linux Host Computer

The Linux Host Computer runs a multitasking operating system called TimeSys which is a Real-Time version of Linux. This multitasking operating system allows task priorities to be set in parallel, ensuring that important tasks are performed in a deterministic time frame. The Linux Host Computer manages the proDAS Test Results and Sensor Calibration Database (TRSCDB) and executes the proDAS Real-Time data acquisition software (RTE). The Linux Host Computer, networked to all other proDAS computers, interfaces with the hardware subsystems to acquire and process engine test data.

Management GUI Computer

The Management GUI Computer is a PC-based computer running Windows. The Management GUI Computer communicates and interacts with the proDAS real-time software, the Test Results and Sensor Calibration database, and individual hardware subsystems. The operator is provided with a graphical interface called the proDAS Management Interface. The functions on the Management GUI Control Panel are point-and-click activated by the operator. Each main function, on the Management Interface, is security controlled and sensitive to the login level used by the operator. An operator will use the proDAS Management Interface in the preparation, testing, and reporting stages of an engine test.

The Management Interface allows the operator to:

  • select the test configuration
  • prepare proDAS for an engine test by performing pre-run checks
  • use the configuration editor for creating and editing channel definitions and test configurations
  • calibrate channels
  • add or delete user accounts and change passwords
  • control test sequences and data recording
  • process and report test data

Real-Time Display Computers

The Real-Time Display Computers are PC-based computers running Windows.

The Real-Time Computers display the engine parameters that are being scanned and recorded. The engine test operators and recorders are provided with a configurable number of Real-Time Displays. These GUIs (screens) allow the operator to:

  • monitor the engine test data
  • display operator pages
  • create and modify operator pages
  • setup transient log definitions
  • monitor the engine test
  • record data
  • view the event handler window
  • playback previously recorded engine test data
  • process and report test data

The hardware subsystems provide the proDAS Real-Time data acquisition software with a digital representation of the signal being measured by the hardware subsystem. Each hardware subsystem communicates with the proDAS Real-Time data acquisition software through a dedicated driver that is compatible with the protocol of the associated hardware.

Software Subsystems

In addition to the hardware subsystems identified in Hardware Subsystems, the proDAS System also contains a number of purely software based subsystems.

These software based subsystems provide additional functionality and support features that are available in the proDAS Real-Time data acquisition software. proDAS uses these software subsystems to perform computations, time-stamp data logs, and allow external programs access to data values residing in the proDAS current value table (CVT).

The software subsystems that can be handled by proDAS include:

  • Calculated Channel
  • External Hooks
  • Internal
  • Fullset
  • Log Playback
  • Replay
  • Data Server
  • Limit and Actions
  • Network Data Distribution Service (NDDS)

The NDDS subsytem type is added via the Configuration Server initialisation file.

Configuration Database

The Configuration Database contains all configuration data of the proDAS system which can be changed by the user, except the calibration data.

The Configuration Database is based on XML files in a directory structure located on the Management GUI Computer. proDAS components are reading and writing the contained data via the Configuration Server.

The Configuration Server gives access to the complete database via its automation interface.

The directory structure of the Configuration Database reflects the Configuration Hierarchy. The root directory is defined in the initialisation file of the Configuration Server. The sub-directories EngineClasses and TestCells of the root directory do not contain any files, since they do not correspond to a configuration. Each of the other database directories contains an Configuration.xml file containing the meta data of the configuration and may also contain a file for each data category as listed in the following table - except for the engineering units which only appears in the root directory.

Test Results and Sensor Calibration Database (TRSCDB)

The proDAS System uses an ORACLE database as a primary storage location for data. The database is a three-dimensional series of related tables that have been structured to accommodate calibration data and engine test data.

Data is placed into these structured tables by various applications and utilities. The proDAS Real-Time engine stores data in the database during an engine test and the proDAS User Interface interacts with the database during calibration activities. For more information, see Test Results and Sensor Calibration (TRSCDB) Database.

Real-Time Data Acquisition Software

The proDAS Real-Time Engine (RTE) is a real-time data acquisition engine that monitors and controls the data acquisition process. The RTE is the core of proDAS and is isolated to provide a higher degree of reliability.

The proDAS Real-Time Engine software runs on the Linux Host Computer and communicates with the Management GUI Computer. The proDAS RTE:

  • monitors the operating conditions in real-time during the test
  • acquires data from the engine sensors and the test facility sensors
  • stores the acquired steady state (fullset) data in the TRSCDB
  • stores transient data on the Linux Host Computer hard disk
  • monitors critical channels and raises and handles alarms as necessary if preset channel limits are exceeded (alarms can trigger actions such as: save a critical log, set/clear discrete channels, etc.)

During an engine test, various sensors, located on the engine and in the engine test facility, monitor such operating parameters as pressures, temperatures, and frequencies. The sensors convert the physical measurements into electrical signals (engine test data), which are scanned by proDAS and placed into a predefined area of memory, on the Linux Host Computer, referred to as the Current Value Table (CVT). The Current Value Table holds instantaneous scanned values, identified by channel name, for every input and output channel in the system. The data stored in the CVT is used for screen updates, logging, cross-channel calculations and alarm monitoring.

Real-Time Engine States

The RTE transitions through the various states during a typical engine test. The Real-Time states are determined by operator input from the Management GUI Computer. The actions that can be performed by an operator are dependent on the current Real-Time state. The Real-Time Engine states are:

  • initialisation state
  • no_state
  • configuration state
  • idle state
  • start_scan state
  • scan state
  • stop_scan state
  • close state
  • abort state

Initialisation State

The initialisation state is referred to as a transition state and occurs when the RTE is first started. During the Initialisation state, the RTE parses the initialisation file and initialises its internal data structure. This transition state is very quick and completely transparent to the user of the system.

No_state

Once the RTE finishes its initialisation, it is the no-state. While the RTE is in this state, it waits for further instructions from the Management Interface.

Configuration State

When the Operator of the system has selected and applied an engine configuration, the data is downloaded from the configuration database to the RTE and the RTE enters the configuration state. The Unified Event Log records and displays all messages as each subsystem is configured.

If system errors occur during the configuration state, the RTE reports the error and attempts to correct it. If successful, the configuration process continues. On successful completion of the configuration process, the RTE enters the idle state.

Some subsystems may fail to configure but the RTE will still continue configuration. The RTE reports the failure in the Unified Event Log, creates a report file (if applicable), puts the subsystem off-line and continues the configuration process.

Idle State

The idle state is reached when the configuration has been completely downloaded to the RTE. The operator is presented with the message "RTE Reference Resolution OK" in the UEL. In the idle state, the Linux Host Computer is not scanning data but has prepared all of the subsystems to acquire data. When the RTE is in the idle state, the user can:

  • start an engine test
  • edit operator pages
  • edit transient log definitions
  • play back saved transient and critical log test data (steady-state and transient) from the current or previous tests
  • exit proDAS

The user's ability to perform some of these actions will depend on the access level assigned to the user name.

Start_scan State

The start_scan state is a transition state that occurs when the Operator has selected the "Start Scan" button on the Management GUI Control Panel. During this state, the RTE prepares the scan threads to acquire data from each subsystem.

Scan State

In the scan state, the RTE monitors the engine test, acquires and logs data, and displays the test data. When the RTE is in the scan state, the operator can:

  • carry out test procedures
  • monitor and respond to alarms or system events
  • take fullset recordings and save them to the database
  • save critical logs
  • start and stop transient logs
  • play back saved critical logs and transient logs
  • print a hard-copy snapshot of the current values of all of the channels being monitored
  • add comments to the Unified Event Log (UEL)
  • display and modify operator pages
  • calibrate channels

Stop_scan State

The stop_scan state is a transition state that occurs after the Operator has selected the "Stop Scan" button on the Management GUI Control Panel. During this state, the RTE closes the scan threads to each subsystem.

Close State

The close state occurs when the Operator selects either the De-configure proDAS or Exit proDAS from the File menu of the Management GUI Control Panel. The RTE closes progressively until the application has been exited completely.

Abort State

The RTE encounters the abort state when the operator de-configures proDAS during the configuration process.
