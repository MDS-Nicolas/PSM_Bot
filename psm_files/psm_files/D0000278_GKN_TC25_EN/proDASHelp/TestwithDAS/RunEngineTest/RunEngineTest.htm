Running an Engine Test

Overview

This module is designed to be used as a reference guide in the everyday work of the test cell operator. The different operator functions that can be performed while proDAS is scanning and acquiring data are explained through step-by-step procedures.

Before an engine test can be started and data acquired, there are several set up tasks that may be required to be performed. The following is a list of preparation tasks that might apply and the associated sections that describe the desired task:

  • Performing Calibrations - see Sensor Calibration Application (SCA) and Sensor Editor for details on how to perform calibrations on VXI channels
  • Performing Thrust Calibration - see Thrust Calibration
  • Setting up Data Recording - see Setup Data Recording for details on how to set up data recording functions such as transient logs
  • Setting up Test Procedure Scripts - see Script Editor
  • Creating Operator Display Pages - see Creating and Modifying Display Pages

After all the appropriate tasks have been completed proDAS must be configured with a master configuration. The operators are required to enter test information including test name, engine serial number, and the operator’s names into the Test Information Page. Please see Starting, Configuring and Stopping proDAS for more information on the configuration process.

proDAS gives the operator the ability to run various pre-run checks prior to acquiring data. These tests may allow the user to perform functions such as confirming channel readings are within the appropriate ranges, and zeroing temperature and pressure channels.

The execution of the pre-run checks depends on Customer hardware and procedures. For details on the pre-run checks, see Instrument Controls.

Please refer to the "Home" section of the Docstation for details on your applicable pre-run checks.

proDAS provides several user configurable displays that can be used to view the current data during an engine test, including:

  • Real-Time Display (RTD) Driver
  • Spotlights
  • ARINC Display (available only if ARINC is configured)

proDAS performs logging of all system event messages, including operator interactions, alarm conditions, warning messages, and error messages using the following:

  • Unified Event Log (UEL)
  • Alarm Summary Display (see Interpreting Alarms and Monitoring)

During an engine test, most of the testing tasks can be performed using the Test Page. Please see Opening the Test Page for further details.

proDAS has the ability to run predefined engine test procedure scripts that instruct the operators on how to test the engine as well as automatically record data such as fullsets or logs. Please see Selecting and Running a Test Procedure for further details.

If required, the operators may also record test data independent of an engine test script. proDAS can record the following:

  • Transient Logs
  • Fullsets
  • Critical Logs
  • Continuous Logs (if configured)
  • Test Comments
  • High Speed Logs (if HSV hardware is available)

During an engine test the operator has the ability to modify the following channel parameters:

  • Set/Redirect Channel Value
  • Set Channel Quality
  • Set Channel Limit

proDAS gives the operator access to several reports that may be useful during an engine test. These reports are accessed through the Real Time (RT) Reports GUI.

proDAS has a built in Timer that operators can use during the engine test. Please see Using the Timer Feature for further details.

Selecting a Real-Time Display

The Real-Time Display System is used for viewing engine data in real-time. It consists of the Real-Time Display Editor and Real-Time Display Driver.

For more information on the Real-Time Display Editor, see Real-Time Display Editor in Creating and Modifying Display Pages.

Real-Time Display Driver

The Real-Time Display Driver (RTD Driver) displays information during the execution of a test run and allows the user to control test bed and test objects (an engine or component such as a compressor). It displays pre-configured display pages and Spotlight Sticky's.

The user can execute multiple instances of the Real-Time Display Driver on the same computer at the same time. This is particularly useful if multiple views or multiple monitors are used on the same PC.

When the RTD Driver is closed the Spotlight Sticky definitions are saved to file and the existing Spotlight Sticky definition files are deleted. When the RTD Driver is re-opened the most recent Spotlight Sticky's are automatically loaded.

Performance can be impacted if too many displays are selected.

The RTD Driver comprises the View Test Page window, the Control View and the Automation Interface.

For detailed information on the COM interface(s), refer to ICD78026.2822 Real-Time Data Proxy Server.

To pause and zoom an entire Operator Display:

  1. Select Execution/Pause or the pause button .
    The display page stops updating. Data required for graphical displays, such as YX Plots and Strip Charts, are buffered.

  2. Select the zoom buttons to zoom in (+), zoom out (-) or reset (=) the display page.
  3. Select Execution/Pause or the pause button .
    The display page resumes updating.

    The user can pause individual RTD Active X Controls using the context menu of the desired RTD Active X Control.

  1. Select New Group.
  2. Specify a string for this subset of fullsets.
    The string is displayed as part of a legend in the diagram of graphical RTD Y/X and/or Profile Plot Controls on the display page. By default (i.e.: if nothing is entered by the user in the respective input field), this string contains all the information specific to the currently used configuration (i.e.: engine type, engine standard, built, customer, etc.). In addition, the shape and colour of the marker symbols, indicating the fullset data points on the diagrams, may be specified.

    3. The colour selected for these marker symbols is applied to both the RTD Y/X and Profile Plot Controls, their shape is only applied in case of RTD Profile Plot Controls. In RTD Y/X Plot Controls, the shape of the marker symbols for the overlaid fullset data is adopted from the corresponding online data points.

  3. In order to access fullsets available in the database, several items displayed in the tree control list on the left hand side of the dialogue have to be unfolded by clicking on them.
  4. Mark a specific test (lowest level of the configuration tree) to display all fullsets taken during this run in the Selection Options spreadsheet to the right of the tree control.

In this filter settings dialogue, not only can certain value ranges for specific channels be defined but also the conjunctions and disjunctions of pairs of those value ranges.
    After the changes made in the filter settings dialogue have been applied by closing it again, the filter criteria may be applied to the list of selected fullsets by selecting the button on the Selection Options toolbar. Then, only those fullsets for which the filter criteria are fulfilled are displayed in the list.
    Clicking on the button selects all fullsets currently displayed in the list. Specific fullsets may also be selected or deselected by clicking in the corresponding check boxes in the fullset list. The button deselects all fullsets in the list, while the button inverts the selected and deselected fullsets.

  5. Using the Spotlight Feature

Spotlight Sticky's are small windows created during an engine test which display current values for some channels. They are useful for troubleshooting or monitoring a channel not appearing on one of the currently selected Real-Time Displays.

When the RTD Driver is closed the Spotlight Sticky's are saved to file. When the RTD Driver is re-opened the Spotlight Sticky's are automatically loaded.

When setting channel values through spotlight sticky's, a warning message appears in the UEL if the value is not appropriate for the channel. If the value is appropriate, a message appears in the UEL showing the update is successful.

Unified Event Log (UEL)

The Unified Event Log (UEL) is responsible for tracking all events that occur while proDAS is running. All messages from the Real-Time Engine are displayed in the UEL Display window, along with filter masks, are stored in a single event log file. Text messages are written to the UEL whenever an event occurs (e.g.: starting test monitoring, taking a fullset, or registering an alarm).

A new event log is created and started whenever proDAS is reconfigured.

Event logs are saved to the hard disk, of the Host Computer, in the file:

/users/RTE/logs/enginename/testname/event_logs/elYYMMDDHHmmss

where:

  • enginename is the name of the test engine
  • testname is the name of the test record
  • el identifies the file as an event log
  • YYMMDD is the year, month, and day on which the log was opened
  • HHmmss is the hour (0-23), minute, and second at which the log was opened

The events displayed on the event log are stored in a text file. The event log can be stopped at any time, and all statements will be saved to the text file.

The messages written in the text file support non-English characters.

Unified Event Log Display

The messages stored in the event log are displayed in the UEL display. proDAS may open several UEL displays that contain filters that determine the events that appear in the window. New messages always cause the UEL display to scroll down to the last entry in the event log.

Interpreting Alarms and Warnings

During an engine test, an operator will be notified, by an Alarm or Warning, when the value of a monitored channel is beyond the safe operating threshold. The operating threshold is determined by the limit value assigned to the channel(s) in the configuration. proDAS indicates the channel that caused the alarm or warning, the initiating subsystem, and the type of alarm in the Alarm Summary Display Window. A warning or an alarm can trigger a subsequent predefined action, such as changing the value of a discrete channel, recording a fullset, or starting a transient log.

For detailed information on defining alarm conditions in the test configuration, see the Channel Editor.

Warnings indicate when a channel is exceeding its defined normal operating range. Warnings do not trigger an audible alarm and may or may not require acknowledgment by the operator. A warning is indicated by:

  • displaying a message in yellow in the Event Handler window and in the Alarm Summary window
  • recording the warning message in the Unified Event Log (UEL)
  • by changing the operator page objects that represents the channel change to yellow

Alarms indicate when a channel has surpassed its defined extreme value and operator intervention is required to correct the problem.

An alarm is indicated by:

  • displaying a message in red in the Unified Event Log window indicating subsystem name, channel name and alarm type (HIHI, LOLO or DISCALARM)
  • displaying the channel in the Alarm Summary window with its limit value and current recorded value
  • a message recorded in the Unified Event Log
  • the colour of any display object showing that channel will change to red
  • an immediate audible alarm (buzzer) is sounded (if so equipped)

All alarms must be acknowledged by the user. If the audible alarm is triggered by more than one alarm, then each alarm must be acknowledged. However, the audible alarm is turned off after the first acknowledgment.

Warning and alarm types that are processed by proDAS are:

  • high alarm (HIHI) - red
  • high warning (HI) - yellow
  • low warning (LO) - yellow
  • low alarm (LOLO) - red
  • rate of change (ROC) - yellow
  • blue discrete info message (DISCINFO)
  • red discrete alarm (DISCALARM)
  • invisible discrete alarm (DISQUIET) provides no audible or visible signal that the limit has been violated

A channel that has one or more sliding limits can have limits that contradict each other. For example, if channel B has a LO limit defined by channel C as well as a fixed HI limit of 150, it is possible for the LO limit to be greater than the HI limit. These contradictions are resolved according to the following rules of precedence:

  • LO limits can be overridden by any other limit
  • HI limits can be overridden by LOLO or HIHI limits
  • LOLO limits can be overridden by HIHI limits
  • HIHI limits can never be overridden

Alarms Summary Display Window

The Alarm Summary Display window provides a list of the proDAS channels as the channels transition from normal to warning and alarm states. The Alarm Summary Display window can be moved and resized by dragging the window borders.

The Alarm Summary Display window is displayed when Start Scan is selected on the Management GUI Control Panel.

The Alarm Summary Display window:

  • displays a list of proDAS channels in alarm and warning states, including limit values, current values, minimum or maximum value reached while in alarm, and the maximum deviation beyond the limit state
  • displays channels in an alarm state (HIHI, LOLO, or DISCALARM) in black characters on a red background and displays red characters on a black background when the alarm is acknowledged
  • displays channels in the warning state (HI or LO) in yellow characters on a black background when unacknowledged and displays black characters on a yellow background when acknowledged or if proDAS is configured so as to not require the acknowledgment of warnings
  • displays an ordered list from newest to oldest, with channels in an alarm state followed by channels in a warning state
  • inserts new alarms at the top of the list
  • inserts new warnings at the top of the warning list (just after the oldest alarm). As new messages are added, channels below the insertion point are moved down, with any channel at the bottom of the window disappearing
  • upon receipt of a new alarm, the display is brought to the front of all running windows
  • displays a channel in the Alarm Summary Display window only once, regardless of how many times it transitions between alarm and warning states

When idle, the Alarm Summary Display window periodically sends a heartbeat message to proDAS. If a preset number of consecutive failures is encountered, the Alarm Summary Window checks for the presence of the RTE. If the RTE is not found, the Status bar will show the RTE connection as being lost. If the RTE connection is lost, the Alarm Summary Window will remain open until it is closed by the operator or by the Management GUI.

During scanning, the Alarm Summary Display window periodically queries proDAS for the current channel values. If a preset number of consecutive failures is encountered, the Alarm Summary Window checks for the presence of the RTE. If the RTE is not found, the Status bar will show the RTE connection as being lost. If the RTE connection is lost, the Alarm Summary Window will remain open until it is closed by the operator or by the Management GUI.

The alarm summary is sorted by red Alarms and yellow Warnings and is then sorted chronologically within each group. The user cannot select a specific alarm or alarms from the list and cannot change the sort order of the list.

Recording Transient Logs

A transient log is a record of selected channel values over a predetermined period of time. Many different transient logs can be defined, but only four can be recording data simultaneously (not including the critical log or continuous log). More than one instance of the same log can be running at one time. When multiple versions of a log are running, they can be stopped and saved individually.

Transient Logs can be started:

  • from the Logs Panel on the Test Page
  • automatically, as alarm conditions in response to a limit violation

A running transient log can be stopped at any time. When a running log is stopped, its contents are saved to a transient log file. If a transient log is not stopped manually, it runs for its defined duration, stops and is saved by proDAS. Two or more instances of the same log can run at the same time and each instance can be stopped (saved) individually.

Transient logs can be started and stopped automatically when an alarm occurs. Alarm conditions that start and stop transient logs are defined, for a channel, in the configuration for the current test. When a running transient log is stopped by an alarm condition, all instances of the running log are stopped, but only the first instance of the log is saved.

For detailed information on defining alarm conditions in the test configuration, see the Channel Editor.

For additional information on Transient Logs, see Setup Data Recording.

Starting a Transient Log from the Test Page

  1. Using the Logs Panel (below) select a log (e.g. VibAccel) from the Log Definitions field.
  2. Select the right arrow button or double-click on the log name to start the selected (highlighted) log in the Log Definitions field. The selected log is displayed in the Running Logs field.
    The transient log begins recording and a message is written to the UEL. The transient log recording continues until the time duration expires or the transient log is stopped manually.
    A message indicating the start time, and the name of the transient log is displayed in the Unified Event Log.
    3. The Comments drop-down menu and panel show any comments relating to the Log Definition. To edit comments or add new comments, see Inserting Operator Comments.

Recording a Fullset (Steady State Data)

Steady-State fullsets are time averaged data from each of the channels measured by proDAS for the current test (all of the channels in the current configuration). For example, if the time duration of the steady-state fullset is one second, and the channel data is sampled at a 10 hertz scan rate, then the 10 samples taken during the fullset period will be averaged to a single value. The value will be stored in the proDAS database under the current test name as the channel value for the recorded steady-state fullset.

A steady-state fullset can be recorded at any time during an engine test. The operator can specify:

  • the time period (i.e.: 0 to 300 seconds) over which the data is averaged
  • a Test Step to identify the fullset
  • an additional comment

For information on retrieving and reviewing fullset data, see Viewing Fullsets and Events using RAVE in Viewing Engine Test Data.

For additional information on Fullsets, see Setup Data Recording.

Continuous Logging

Continuous Logging is the implementation of a continuous recording and playback utility that automatically records hardware channel data whenever proDAS is scanning. A trigger channel is used to control the starting and stopping of the continuous log. The name of the trigger channel is obtained from the RTE initialisation file. The RTE initialisation file can also be used to exclude specific hardware subsystems from the recorded data. For details, see Continuous Log Inputs.

Up to 20 complete test logs can be written and stored by proDAS. When the number of continuous logs exceeds 20, the Real-Time Engine (RTE) will prompt the user to shutdown and archive; continue and overwrite the oldest log file or continue with the continuous log disabled.

A new log type is defined for the continuous log, called continuous_log, and is stored in the same format as the existing critical and transient logs in the \users\RTE\logs\continuous_log directory. A link to the log file is also found in the \users\RTE\logs\<engine_name>\<test_name>\hist_logs directory.

The channels to be recorded in the continuous log are assembled at configuration time from the configuration data.

If no trigger channel is defined when proDAS stops or starts scanning, the continuous log is automatically stopped and restarted. When restarted, proDAS attempts to append to the already existing continuous log file. When a trigger channel is defined for the continuous log, this channel controls the stopping and starting of the continuous log.

If proDAS is reconfigured with the same configuration and test name, the logs subsystem will verify whether the already existing continuous log file contains the same channels as the current configuration. If so, it will attempt to append the new data to the existing continuous log file.

If appending is not possible because of a configuration change or because the log file is already larger than the maximum log size allowed, a new continuous log will be started.

Changes to the following channel attributes cause a new continuous log file to be created:

      •       critical flag
      •       scan rate
      •       subsystem type
      •       discrete test true/false strings (discrete channels only)
      •       min and max values (float channels only)
      •       engineering unit (EU) (float channels only)

Changes to the following test attributes cause a new continuous log file to be created:

      •       engine serial number
      •       build number

A continuous log event file is used to associate the individual continuous log files for the same test. This file records the start and stop times of each log, and an index into the respective log file for each start or stop occurrence. This information is then used when replaying the log file.

The continuous log obtains configuration information from the LOG_PLAYBACK section of the RTE initialisation file.

There are parameters within the RTE initialisation file that allow the log to continue recording until it reaches the 2 GB maximum size. When this happens, a new log is started with no time delay. These parameters allow a continuous log to not be stopped if it is below the size limit but over the time limit.

Setting the Quality of a Channel

The channel quality specifies the reliability of that channel and can be set by:

  • the operator from the Test Page, on the Management Interface Computer, while proDAS is running
  • the proDAS application while proDAS is running

The operator can mark the best quality value, known as the quality ceiling, that a channel may read.

proDAS sets the quality to GOOD (when the data is sound), SUSPECT (when the data is interpolated or is otherwise suspicious) or BAD (when the value cannot be acquired by the hardware). However, proDAS can never set a better quality than the current quality ceiling for a given channel. For example, if the operator sets the quality for channel A to SUSPECT, proDAS cannot set the quality to GOOD, even if a good value is measured. proDAS can only set the quality flag to BAD or SUSPECT for this channel. The operator can however, modify the quality and set the quality ceiling to GOOD, thus allowing proDAS to set any quality value.

Once the operator sets the quality of the channel, the data is recorded along with each measurement, for example:

  • setting the quality ceiling to BAD causes proDAS to record a data value of -99999, and set the quality flag to BAD
  • setting the quality ceiling to SUSPECT flags to the end user that there is something suspicious about the data, and causes proDAS to record the actual data value and set the quality flag to SUSPECT
  • setting the quality ceiling to GOOD causes proDAS to record the actual value and set the quality flag to GOOD

The quality flag is recorded with the data in any transient log or fullset. To set the quality of a channel:

  1. Select Set Channels from the Test Page.
    The Set Channel control panel, including the Set Quality Ceiling options, is displayed.
  2. Select to display the Select a Channel window. Choose a channel from the list and click Select. The chosen channel is displayed in the Channel Value field, the value for that channel is displayed in the Value field, the quality of the channel is displayed in the Quality field and the ceiling for the channel is displayed in the Ceiling field.
  3. From the Quality Ceiling pulldown, select the quality ceiling for the channel.
  4. Select Set to set the channel.
    The new value is set, and the changes are displayed in the Event Handler window and recorded in the Unified Event Log.

Modifying Channel Limits While proDAS is Scanning

Channel limits are thresholds that determine when a warning or alarm is realized by proDAS. The channel limits are established in the engine configuration that is downloaded to proDAS.

For detailed information on setting limits on channel alarm conditions in the test configuration, see the Channel Editor.

This proDAS feature allows the operator to make marginal changes to the channel limits while proDAS is scanning. The changes are controlled and are kept within an allowable ± tolerance band of the original limit value. The new limit values remain valid until proDAS is reconfigured.

The operator can use this feature when a known channel is very close to the alarm threshold and is cycling in and out of alarm many times per minute.

Only an authorised user may set channel limits.

For details on the Set Channel Limit panel, see Set Channel Limit Panel in the Management GUI.
